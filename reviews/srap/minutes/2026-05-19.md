### BIBO "Contribution" class

- https://github.com/dcmi/dc-srap/issues/100 - Property names for use with Contribution instances (Tom)

What's an affiliation attached to a class and affiliation attached to an agent. Osma suggested using "contribution" in naming. 

Osma: That was one of Tom's original suggestions. I liked that one the most. These are technical and are scoped to the contribution class/entity; makes it clear that these are scoped to the contribution.

Tom: "has" is vague.

Karen: People would see the label, not the name. This is a database for people to find things. 

Tom: User view out of scope

Karen: Created [issue 101](https://github.com/dcmi/dc-srap/issues/101). We will need to provide documentation and there are things unclear without docs. Eg: we have multiple data elements for persons and contribns. dct:creator/contrib in the "main shape" and use of Contribution shape. Will be confusing unless we address in docn. Worst case: sometimes name goes into dct:creator, sometimes dct:contrib. Situation must be explained.

Tom: Could use both patterns: directly stating the author using dc:creator, and indirectly through a Contribution object. For example, use a dc term like dc:creator in the object position of hasRole (treating it like a skos:Concept) and have property chain axioms so that the direct and indirect expressions can be derived from one another

Osma: Gluing things together with OWL axioms does not solve a big problem but connects the "verbose" with "simple" form. Could add property chain axiom. Not ffamiliar with details.

Niklas: General-purpose terms like "date" are applicable to many types of resource. Design of BIBO is clearer. Was leaning to "contribution" names. If we accept the dedicated design where the properties for Contribution are prefixed with "contribution", we could also define a generic affiliation for agents.

Tom: Yes, if we define a dedicated contributionAffiliation, we should probably also define affiliation for agents.

Karen: Didn't we define these since we needed to capture the context of the contribution?

Tom: We did, but there is possibly also the general case of affiliation outside of the contribution.

Karen: How would the second case be defined in the DCTAP? In the original case, as Juha described it, the person would have a role and an affiliation.

Tom: Before the UB started, there was a proposal for qualifiers. There was also a new type of qualifiers called components, as part of the description (such as affiliation). The idea didn't pass the review. In comparison with the issue from 2021, Jan Ashton pointed out the decision for datacite from 2014, where a child element to creator/contributor elements supplied affiliaiton, like the components.

Osma: We are circling back to the original problem. A general affiliation property might be useful, but its simple use has presented us with this problem of not being associated with the context. In the original XML-style designs, this was not designed to work over time. So I don't think there is a requirement for having an agent-specific (or general) affiliation property.

Tom: Do we want to take the temperature of the group, to see which terms we think we need? The definitions (e.g. their "meta" nature) should make their respective usages clear.

Phil: If we create this or these properties, what namespace should they be defined in? In DC terms, the domain may be left blank; in BIBO properties have clear domains and ranges.

Niklas: Understood would be in BIBO. If we see no requirement for "affiliation" property could make later. No general need for such.

Karen: If we add these to BIBO, how do we define these to always be in the Contribution shape? I think in BIBO it would be OK to have affiliation, and in SRAP we define a shape and have the label there be "contribution label".

Osma: Since these are meta-properties only to be used with Contribution, in BIBO this should be the domain (or domainIncludes).

Niklas: Yes; and these should probably be called contributionAffiliation etc, and use domain.

Karen: A Shape is not the same as domain. I don't see how domain solves this.

Phil: If you set the domain to Contribution, you know that thse properties are only used with Contribution entities. The shape would enforce the use.

Karen: The property descriptions . We still need the DCTAP to describe the usage. These properties could be more general than that, and used in other ways.

Niklas: Definition of class, added to BIBO, should be clear enough to be isomorphic to shapes. These properties could be defined within domains and ranges in the BIBO style. 

Tom: Can we have a strall poll on a comment in https://github.com/dcmi/dc-srap/issues/100?

Karen: BIBO has https://dcmi.github.io/bibo/#event:agent .

Karen: It is about the scope. I don't think limiting this to contribution doesn't make it as broad as the rest of BIBO, and definitely not as the DC terms.

Phil: There may be a potential need for the general terms, but if we define these with this specific meaning we could add those later if needed. 

Niklas: Agree with Phil's thinking - if need, would not want to define in way that precludes more general. Against the more general terms but since BIBO really is more structured with domains/ranges - about biblio resources - not properties for other types of things. We are adding with clear scope. Could also make case that general affiliation prop be defined - could define something like `bibo:affiliation owl:propertyChainAxiom ( [ owl:inverseOf bibo:contributionAgent ] bibo:contributionAffiliation ).` So we these clear definitions in BIBO. Could do same for role. There are ways, eg authority records listing roles. Given that BIBO is clear, definitions. 

Tom: Noting that we don't have full consensus, we have a strong agreement on the proposal, and no formal objection to it. The proposal doesn't preclude general terms being added, and we can add the finishing touches later. Can we have someone propose this and hasContribution?
