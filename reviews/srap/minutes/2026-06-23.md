### BIBO "Contribution" class

#### `bibo:Contribution` (issue 108)

`bibo:Contribution`
- https://github.com/dcmi/dc-srap/issues/108#issuecomment-4777812643 -- let's vote on this

Tom: I adapted the proposal to include "context".

Karen: There is something missing to make it clear.

Argie: The counterpart of role perhaps?

Karen: You don't have the role of an intellectual contribution. I'd suggest the type of the contribution, and the institutional context. If you say it with adjectives and then a noun, they all belong to the noun.

(Some wordsmithing to reach clearer consensus on the formulation. We eliminated "institutional", to put in a comment exemplfying more concretely.)

Karen: The creation of the resource is nowadays considered more widely, including all the scientific work.

Tom: To be clear, we're voting on ContributionContext.

__DECISION__: https://github.com/dcmi/dc-srap/issues/108#issuecomment-4780232901

> bibo:ContributionContext
> 
> Definition: The nature and context of an agent's intellectual contribution to a resource.
> 
> Comment: Used for the role and affiliation of a given agent who contributed to creating the resource.

#### `bibo:contributorRole` (issue 106)

`bibo:contributorRole`
- https://github.com/dcmi/dc-srap/issues/106

Karen: I worry that people not as aware of SKOS might believe that resources linked to *must* be typed as skos:Concept. I thus prefer not to state this. We'd expect people to use these terms in profiles, and there they can define what they need.

Tom: Any referenced role here can be considered a SKOS Concept without ontological harm. It seems to be to be an elegant way to underline that here  something denoted by a URI denoting a concept is intended. I'd even go further and say that the range is rdf:Property AND skos:Concept. This enables further reasoning, such as chain axioms and dumbing down to direct properties.

Karen: I'm not sure I follow. I can see that you're on to some algorithmic use for interoperability, but I don't expect the audience to see this. Also, some uses don't use MARC relators, but e.g. textual terms.

Niklas: I am in favour of the use case of using actual RDF properties here and I would interpret labels as SKOS concepts and would be content with that.

Karen: But what about people just using text (literals).

Niklas: Even if there is just text, there is some notion of a role / concept in the background.

Karen: But how does that work in practice? This came from the SRAP work and we are going beyond what was initially desired for SRAP. Initial idea (by Juha) was that all of these properties would be in Dublin Core and therefore all would use rangeIncludes. By bringing this into BIBO we have kind of changed the nature / ambition of the use case. We are no longer just addressing just the SRAP use case. I do not feel comfortable about that.

Niklas: By defining some more structures we can still cater to the original use case. There IS a difference between the less strict DC terms definitions and BIBO which enforces / enables (?) some error correction.
If we define the range to be skos:Concept, we force a literal to be a SKOS concept ... We still have to infer a lot of context, there is a lot of imprecision.
By taking a range of dct:Agent for role we have already accepted that and we should continue that pattern here. Using BIBO here still fits the SRAP use case but adds some requirements on the side of the data users. 

Karen: If we do that we will need a very clear document that explains this kind of sulotions for people who are not deep into RDF. Give examples and explain how to use this with just textual data.

Phil: The end user need not see this difference -- the developers of the interfaces need to know, and we do.

Karen: The document would bridge the gap between people still using the DC terms methodology with what we are now putting into BIBO. Technical people should not be fighting with input people but technical people should make the most of the details input people are able to provide.

Karen: We cannot use rangeIncludes in BIBO, we have never done that.

__DECISION__: https://github.com/dcmi/dc-srap/issues/106#issuecomment-4780420910

> bibo:contributorRole
>   definition  The nature of the agent's contribution to the resource.
>   domain      bibo:ContributionContext
>  range       skos:Concept
>  comment  Values need not be formally defined as SKOS concepts; it is enough that they be inferrable to be such.

Karen: We need a document to address the concerns we discussed about input vs. technical considerations, and general expected usage.

Tom: It's a great opportunity to have a wider message about the various possible ways to express your information. We don't need a direct action now; once we're done with these details we can take a step back and consider what's missing from the message; and target different audiences.

Karen: Yes, both for input people and developers too.

#### `bibo:contributorAffiliation` (issue 105)

`bibo:contributorAffiliation`
- https://github.com/dcmi/dc-srap/issues/105

Karen: I think we need to start with the institutions rather than the agents.

Argie: The use of the definite article is a bit confusing.

Karen: I can't imagine we would have this without an agent in the shape. The whole context is focused on the agent. The affiliation would be odd without the agent; the affiliation and role are optional pieces to the agent.

Phil: From schema.org: sdo:affiliation An organization that this person is affiliated with.

Tom: We may have multiple affiliations here?

Phil: I suspect so.

Karen: The role is rarely given, the affiliation more often so; usually it is associated with the document.

Argie: We just submitted a document where two insitutions are used for each professor.

Karen: We have not restricted this in SRAP either.

__DECISION__: https://github.com/dcmi/dc-srap/issues/105#issuecomment-4780511293

> bibo:contributorAffiliation
>   definition  An institutional or group affiliation of the agent.
>   domain      bibo:ContributionContext
>   range       dct:Agent

#### Dumb-down of `contributorRole` (issue 95)

Tom: Before going further, I think we could close issue 95 about dumb-down, if there is no interest in that. The idea is that since the MARC relators are both skos:Concept and owl:ObjectProperty, you can also use them as direct properties. This issue is about entailing these from the constructs we define. If contributionRole were to be defined to take this range, we would fulfil this issue. You can still _use_ this capability, it just wouldn't be an open issue on SRAP.

Discussing https://github.com/dcmi/dc-srap/issues/95

Niklas: If the issue is to have two ranges, we can close. That said, could be useful to infer direct relationship from contribution context.

Tom: I think we're talking about defining BIBO. It may be useful to have a note to use this structure to point out that you can derive the simple relationships from.

Karen: I see a nice article that could be written for users of these structures with, e.g., MARC relators.

Tom: We're working on this in relation to a presentation, with more emphasis of different structures. This is not on the critical path for SRAP, but we can revisit it.
