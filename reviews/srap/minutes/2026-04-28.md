### BIBO "Contribution" class

- https://github.com/dcmi/dc-srap/issues/94 Contribution class
- new issue: https://github.com/dcmi/dc-srap/issues/97

ACTION 2026-04-14: Nishad and Karen to visualize her examples.
- DONE -- see: https://github.com/dcmi/dc-srap/issues/98#issuecomment-4335516135 
- also see: https://github.com/dcmi/dc-srap/issues/98

UPDATE from the UB chairs (Tom): consolidated proposal to solve issue #94 (and consequently, #89, #90, and #84)

Tom: issue #94 has become very long so I created a new issue (#97).
I also created a new issue #98 for discussing tools for visualizing SRAP (and Nishad has already reposted his comments over there).

Tom [shares screen]: I listed all issues related to this that we have closed. Two issues remain open (#95 and #96) which we should look at once we have resolved #97. 

Tom: new model "April model" (AM). Contribution associated with Agent, Role, Affiliation. We also want to support the direct DC relation style.

Tom: However, I actually want to argue against AM. What is nice about this is that  it is aligned with Bibframe and PROV-O. However, ... [see https://github.com/dcmi/dc-srap/issues/97#issuecomment-4335908563] (one example, five options to solve it). One option relying on RDF 1.2 which is not officially out yet.
Nice about AM: everything is packaged up in one Contribution package.

Tom: My question -- is that an appropriate depiction of where our discussion was headed up until now?

Niklas: From my perspective, yes, this is the model that I had in mind. I also see how Tom's example points out a problem. Two remarks: Bibframe says that a contribution should have (only) one person and (I think) one role. PROV-O is quite closely modeled around classic way of reification. I am also invested in RDF 1.2 and really hope that that will be used to express complex relationships like this one. Also, SHACL 1.2 has structures for expressing expected vs. optional annotations.

Tom: yes, those would be candidates for an extension of DC-TAP

Tom: issue is not joint contribution, that is an edge case. But it made me realize that role more appropriately belongs to a BIBO contribution object than to an affiliation. Also, Karen once said that "contribution type" might be a more appropriate name than "role". Role is closer to resource, and affiliation is closer to agent -- should they be part of the same object?

Niklas: Attribution is about provenance. Could we even use dc:type with a MARC relator concept (to be parsimonious with the terms)?

Phil: quite intrigued about what Niklas says about attribution, feels like that is what SRAP was about in the first place?

Karen: People are comfortable working with pseudo-instance data. I am aware that soem of this can be solved using RDF 1.2 which we have not tested yet in a DC-TAP, we would have to do that.

Tom: true but even using the more traditional modeling tools -- how many context-dependent infos need to be described in SRAP? is probably more complex to have to separate objects instead of a combined contribution object; I like the new model instead of modeling a new class Contribution. Are we comfortable with conflating affiliation and role in one object?

Karen: we are aiming at academic publications, often multiple contributors; we often had to cut off at 99 contributors, so multiple contributors with unclear contributions is the default

Niklas: instead of using bibo:contribution use bibo:attribution would that improve things?

Karen: no, worse -- either we are gonna have a whole lot of contribution shapes in the data, or one with a whole lot of contributor names and it is unclear what they contributed; wonder how that will look to the people creating the data? can be flexible, be put onto a form etc. -- we will have to think about that as well.

Phil: important distinction between many individual contributions on the one hand and corporate (joint, inseparable) contribution like in Tom's example; and then what about one person's affiliation that is not true for the other? choice: do you treat them as a single contributor? then they don't have that affiliation! also for example the data management team, etc. The joint team does not have the affiliations of the individual team members.
Again, Karen, why would bibo:attribution be worse? To me that would fit a little bit more naturally.

Karen: It depends on point of view? Are you saying these people contributed, or are we attributing this to these people? I prefer active view of contributors. We can call things whatever we want in interfaces and then call it differently in the model behind the scenes.

Niklas: I think point of view matters. If we would call attribution then we would model the data ... I would not encourage the use of RDF 1.2 in cases where we can model more naturally. Annotations are useful if you have a system already in place and need to add information. I agree with Phil's comment about contribution of teams and that it solves the affiliation problem. Also interesting option to have flat lists attached to each contributor with more details about them.

Karen: [shares screen] Finnish repository, vocabulary they are working with -- they also have "groupauthor" so they have already solved this, and we have nothing corresponding in SRAP yet. I also should mention that they are using DC 1.1 with the extensions, so moving them up to DC terms would already be a big move.

Niklas in chat: Group author could be expressed as:
```
<resource> dct:contributor _:g .
_:g1 a foaf:Group ;
  dct:type lcrel:trl ;  # trying out just dct:type
  foaf:name "The Translator Team" .
```

Tom: but they are using XML schema so we don't now what their interpretation looks like.

Karen: It speaks to their point of view and how far they have not gotten in terms of updating their metadata

Tom: dc.contributor.affiliation and such -- that is not RDF but one could provide some ideas how that could be converted into SRAP for purposes of interpretability by using e.g. Python scripts. Dot tradition goes back to the 1990s, split off before RDF. I think there is something that SRAP could offer to create interoperability.

Karen: I look at their XML schema and it isn't using the dot notation?

Tom: I was only looking at the specification.

Nilas: But the namespace includes the terms? They don't express anything in the dot notation

Karen in chat: https://github.com/dcmi/dc-srap/issues/96; The flat SRAP that I did and Nishad visualized leaves off quite a bit -- we have to think if we want to add in more
    
Tom: Examples in DataCite are expressed in XML, intention is that creator. ... you are creating nesting, you could output RDF triples that are compatible with other models; would be nice to show some examples how that could be done; bothers me that Finnish repository requirements actually led to creation of SRAP WG -- product in the end should address those requirements in some form.
    
Nishad: If you have DC TAP you can directly export to SHeX and SHACL using Tapir
    
Nilas: can we decide on anything yet? no, too many people missing
    
Tom: using the AM as a reference point, should we use that to clarify and finalize? I am going to add some more issues in the coming days 
    
Niklas: quick temperature check -- option 1 solves the problems, so if we take AM and option 1, is that the general direction we are moving in?

Tom: I see these issues with it (see my comment in issue #97) so I am not wildly enthusiastic about it but I think it can work although it is a little bit funny

### Overview of to-dos for publication of SRAP as as DCMI AP (~30min)

Taking stock of what needs to be done before we can publish SRAP as a DCMI application profile (also see project board https://github.com/orgs/dcmi/projects/4/views/1)

- solve issue #97 -- which supersedes #94 (Contribution class) and associated issues (#89,#90,#84), see above; also decide on the naming of elements
- decide on accessibility properties -- issue #79; separate meetings with Tom, Niklas, Kalli; 
- review additions to BIBO for completeness (issue #91 (and #86))
- approve BIBO 2.0 and update with new properties and classes; ontology or vocabulary?
- flat SRAP (how can SRAP interact with flatter models?)
- review and approve documentation as a whole for completeness and readability (issue #131)
    
Tom: I was hoping we could mostly finish before the summer but we only have two more months and it is a big list

Karen: at some point we need to declare that
    
Tom: yes, (series of) blogpost(s); also working on a paper with Kalli, Niklas and Labra for DCMI26 in Seoul about design decisions
    
Tom: final review of documentation can be done in the fall
