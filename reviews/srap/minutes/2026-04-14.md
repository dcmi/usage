### BIBO "Contribution" class

- https://github.com/dcmi/dc-srap/issues/94 Contribution class

ACTIONs 2026-03-31: 
- Karen to create example of using non-DC namespaces in SRAP
    - see: https://github.com/dcmi/dc-srap/issues/94#issuecomment-4233113089
- Kalli to create strawperson proposal -- done, see https://github.com/dcmi/dc-srap/issues/94#issuecomment-4173380009

Argie: Any status for the TODOs above? The strawperson proposal is completed. And the examples has also been added. Do we intend to vote, or discuss this proposal further?

Karen: I assume it's because we have some doubts. We have examples, but I'm not sure it's the best way to go. I think it works, but I don't think this is in the spirit of either DCTAP or SRAP. They were intended to be a document/conversation between different people; some of whom are not very technical. This contribution with agent won't resonate with people creating simple metadata for repositories. You may be able to make the interface so that it is how people think about the data, but that puts the burden on that, and not on us to promote simpler data.

Argie: Do we have another proposal that can be put side by side?

Karen: In the SRAP github repo there is a proposals folder with various propoals. I still don't know what the goal is for this use of contribution.

Osma: As I see it, the goal is to be able to express the roles and/or affiliations that the agent has had when contributing to the work.

Karen: But I think we have that in the person shape.

Osma: The problem with that is that it is not compatible with RDF thinking. We are only speaking in the context of this publication, and not generally.

Karen: I made an example of that. [shares screen: viewing the srapcontributors.csv example] Person just has the name and identifier, as does the organisation. People are probably going to spend some time in these SRAP shapes, to understand them; they are a bit complex. In the example I use the BIBFRAME properties, to highlight what is going on. It got tricky in relation to current SRAP, where we have an organization and publisher, where it now goes to an agent shape.

Osma: I don't see any reason to use an agent shape. In RDF you may have a superclass, but there is no need to capture that in a concrete TAP shape.

Osma: do you prefer the srapcontributors version?

Karen: Maybe, but what to call it. There is a "box". See Phil's diagram. We still have a person shape and an organization shape. But do you have to go through a contribution? Where do you go for publisher?

Kalli: The other shapes made me pause. We do similar things in the BIBFRAME group. In BF also publication is a nested structure.

Osma: I think what you propose is probably equivalent to the proposal?

Karen: It depends if you can go directly from the resource to the person.

Osma: As I see this, it is an additional layer to provide more information about the contribution.

Karen: It seems a bit tricky to have a middle step that you don't always use. If it was a leaf I could understand it.

Kalli (in chat): For the BIBFRAME interoperability group, we do have DCTAP that use contribution shapes and it is not too difficult for the non-technical staff, though I realize that it is a different set of metadata creators  https://github.com/bf-interop/DCTap/blob/main/Monograph%20DCTAP/Monograph_Work_Text.tsv

Niklas: That is primary motivation for annotations in RDF 1.2. In most data-consuming cases, look at simple relationship. Data structure that has that always is preferable. Tricky when you need to contextualize simple relationship. Tricky in RDF. Rather than remodeling for complex situation, instead attach those as footnotes to simple relationship. I struggle to explain that approach. Intuitively easy on whiteboard, but in triples, cannot rely on implicit understanding. In any structured data, if you have multiple ways of interpreting -- is hard. Defer adding role and affiliation until annotations can be added?

Karen: You could also add the contextual information "later", in the other direction, as Phil suggested (adding the contribution to the person).

Tom: I think this is a case where you don't have to choose. We can have one pattern for the simple case and another for the complex one. When you no longer can nest the information in a flat model, you can go for the nested form. There shouldn't be any technical problem to add a nested model for the explicit scenario. For the simpler shapes the user interfaces don't have to do anything. On the previous calls there seemed to be support for adding both choices to the SRAP model. It may be possible to map the "dotty" notation to the more complex contribution shape.

Osma: We briefly discussed attaching the contribution to the person. We discussed it also with Phil. And you still have to connect the contributon shape to the scholarly resource. Then you'd have a circular reference, which may also be problematic for some systems. The other reason is to follow the BIBFRAME model.

Karen: I disagree that that was what Phil's diagram was about. He did variants for both with diagrams. The first one with the circular structure was more for being able to go directly to the person. Having different patterns in the same DCTAP is difficult to do, since it's a table. If we add the contribution shape to the person shape, you never have to go through the contribution shape to get to the person shape.

Osma: If you don't link the contribution shape to the resource, you cannot find which contribution relates to the resource.

Karen: But you don't need to think about the database in the DCTAP.

Osma: But we already use a contribution shape in our model?

Niklas: Is DCTAP supposed to be designed in way that captures all conceptual information? Some have tried Named Graphs. Hard to differentiate between conceptual model and info structure.

Tom: It would be difficult if we expect the DCTAP to express different description patterns, and if we say use either choice depending on your need. But I think the DCTAP describes a particular context, and it should be able to express the different options within that context, so it sufficiently capture the different needs in a way to reflect the information that needs to be captured. So in effect, a simple SRAP and a more expressive SRAP; wrt. role and affiliation.

Karen: So to me the question becomes: can we create a SRAP that makes (easy) sense for the cataloguers who know the data, and at the same time can describe the additional step (the TAP is documentation). Can we look at how it can transform to the different patterns in the next step, as needed? Rather than doing an RDF design, the people in e.g. Finland are using XML? So the DCTAP is supposed to be format-open.

Osma: That's right; but we're still trying to adapt BIBO so it can be used for this. We still need something that can be used as RDF. I share the concern that the TAP needs to be simple and understandable. But this is just a fact of life; regardless of how we design the boxes, we still need to capture the required information requirements.

Karen: So, then, how do we express the TAP in RDF?

Osma: But the other pieces are already represented; we only need these additional terms.

Karen: Then we need to express the TAP in RDF.

Tom: I think we may use a formal shape, SHACL or ShEx, to express this, since they are designed to express RDF shapes. We then need to express the TAP as either of those, and see how this looks.

Argie: Can anyone take an action to do that?

Osma: We already have examples in RDF and the terms described as RDF; I don't think we need more than that to express what we need.

Karen: I think the diagrams are far from complete. We need something else. When you start do to the examples, you see other things.

Osma: So what would be helpful for that?

Karen: As Tom said, need either a SHACL or ShEx to express these example shapes more fully. It would be a good test to whether it works.

Tom: As long as we use DCTAP in the way it was designed, so that it is compatible with either, if you have a DCTAP, you can generate either, and/or a UML-like representation. E.g. using the Rudoff tool; where you can show using the "or"-case going beyond the conventional UML graphics. If we have both patterns as DCTAP(s), we could generate more complete diagrams. But I don't know what it will give us, more than what we're looking at. So we need a focused look at the simple and complex patterns, rather than the whole TAP. A TAP should express a subset of these languages; so you could generate either and then the UML diagrams.

Karen: [link to diagram](https://github.com/dcmi/dc-srap/issues/94#issuecomment-4078723842) showing the complexity so far, I think.

Osma: So what would an ideal artefact be? A set of SHACL constraints, ShEx schema, or diagrams?

Karen: We need a SHACL or ShEx for all the requirements. I don't know if that would change the shapes, but somebody needs to try that out.

Nishad: [shows a generated diagram from the DCTAP] It is the output of Tapir from the DCTAP.

Niklas: [shows example where extracting information from graph context], not know what role applies to. Losing context; would need to link back to resource. Nesting structure does not necessarily convey what is going on in the graph.
