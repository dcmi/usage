### BIBO "Contribution" class

- https://github.com/dcmi/dc-srap/issues/94 Contribution class

Tom: Looks like we have re-invented the BIBFRAME pattern.

Karen: But BIBFRAME has an authority file. So in bibliographic record, display information, with link to authority file. We decided in SRAP users would not have separate authority files.

BIBFRAME can have Agent and Role within the same shape without confusing that Agent is always in that Role. There is an entire linked record.

Kalli: Listed two examples from Sinopia Editor and LC - both point to real-world. Qst: imagined ORCID IDs - wouldn't they help?

Karen: If we look in SRAP DCTAP: Person has role affiliation name (literal), identifier - four elements. SRAP combines things from BIBFRAME and BF Authority.

Osma Suominen (31. Mar 2026, 16:59)
BIBFRAME Contribution also doesn't normally allow expressing the affiliation of the agent. Although it would be possible to represent that too with a new property

Karen: NAL data re: books. In my SRAP proposals, could separate. Tried DCTAP with Person (like authority record) and Contribution. Take Persons and Organizations, separate identity of Person from the Contribution. Person: name and id. Contribution: role and affiliation.

Osma: Also think good that we arrived at BF. We had concerns about baggage with using directly. But no domain assertion on bf:contribution - so kind of reusable, but some differences. Would we want to have similar mechanism in BIBO, so BIBO gets to same level? Or just reuse Bibframe? Some minor things in BF, but solvable.

Niklas: Modeling, we are converging. One argument for adding to BIBO: we wouldn't be recommending something directly dependent on design evolution of BF. But stable enough? They have removed Domains and Ranges. Not personally like, but appreciate that they help reuse. Would hope no more semantic changes to BF. At NL Sweden. Subproperty chains for relators? Unsure whether consumers of BF expect. 

Osma: We avoided non-DC namespaces in SRAP originally.

Karen: We consciously avoided, but we can show examples. 

ACTION 2026-03-31: Karen to create example.

Kalli: No affiliation property in BIBO - also madsrdf:hasAffiliation we wouldn't need to use. Used at NAL.

Niklas: "Vintage RDF": DC, FOAF, BIBO... Namespace potpourri. Taxing with more than six. Being complementary easier to explain.

Osma Suominen (31. Mar 2026, 17:12)
In my view, BIBO tries to represent academic papers (and of course other things as well). Author affiliations are an important aspect of that, which BIBO doesn't cover currently. If we (in e.g. SRAP) have to turn to BIBFRAME and even MADS/RDF to be able to express those, I think that points to a problem in BIBO.

Niklas: If we define in BIBO, could be subproperties to BF. If BF 4 breaks semantics, could retract equivalence. Hope we will get back to BIBO in other contexts too, while avoiding Schema.org huge scope.
