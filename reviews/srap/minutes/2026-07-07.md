### `bibo:hasContribution`

https://github.com/dcmi/dc-srap/issues/104

By the time we needed to end the call of 23 June, the group was leaning towards a property name of `bibo:hasContribution`. Discussion should pick up on 7 July at https://github.com/dcmi/dc-srap/issues/104#issuecomment-4780652554 and should try to settle the questions of (name), definition, domain, range.

Argie: We were close to consensus. We almost agree on `hasContribution`. Do we agree? ContributionContext is now defined as "The nature and context of an agent's intellectual contribution to a resource."

Tom: `hasContribution` is okay. `hasContributionContext` is long. `contributionContext` would have the same name as `ContributionContext` except for case.

Niklas: `contribution` would be an option, but I'm also fine with `hasContribution`.

Karen: `contribution` alone doesn't say "predicate" - more a noun. `contributes` - no.

Kalli +1 `hasContribution`.

Argie: `bibo:issue`/`bibo:Issue`.

Niklas: `has` added to `hasFormat` - so not too strange. Unlikely to cause confusion. Might want to add classes for publishers, etc - could use.

Kalli: `bf:contribution` - just one property, two classes: `bf:Contribution` and `bf:PrimaryContribution`.

Argie: And the definition? "An agent and the context of their contribution to the resource"?

Niklas: I'm trusting our previous discussion in that regard.

Argie +1

Karen: Suggest adding comments that give example. 

Tom: Approve name and definition and take action for example?

Argie: We have existing issue to revisit BIBO domains/ranges.

Phil: Example note "their" - but agent could be a machine. Put "description of an agent" at the beginning.

Argie: Could we then omit "context"?

Phil: But role and affiliation described.

Niklas: "Description" refers to RDF too.

Phil: So "the context".

Karen: Includes the agent.

Phil: To me, agent is part of the context.

Karen: Will that be generally clear? Agent is the basic thing required here. Affiliation and role are optional. Would not create the node without the agent.

Niklas: "The context of the agent and their contribution to the resource.".

Argie: As long as agent is in definition.

Karen: Agent is required. Context is the role and affiliation.

Argie: Definition does not enforce, but.

Niklas: If you omit agent, no context.

Tom: "The context of the specified agent..."?

Karen: A wrench. My preference would have been for the agent to be primary and, potentially related to that, could give the contribution context. Putting it in contribution makes it unclear.

Tom: Can do in RDF 1.2.

Phil: Original diagrams had arrows in all directions. ContributionContext as a class would stay the same. Karen, your model would point from agent to resource. Not proposing we revisit.

Niklas: Affordance that this definition presents. When RDF 1.2 is out with annotation contribution, can focus on simple relationship. Purpose of definition: give user enough clues to understand structure. Do not need else. Describe context with this object.

Karen: Posted link to diagram that separated context from agent, so that you can easily have agent, always, and then context related to the contribution. Can imagine other ways... 

Phil: "An agent and..." - someone could get wrong idea but can live with it.

DECISION https://github.com/dcmi/dc-srap/issues/104#issuecomment-4780652554 

ACTION 2026-07-07: Someone (?Osma) to write an example for `bibo:hasContribution`.

### DCTERMS Accessibility Properties

Proposed text for public comment
- https://github.com/dcmi/dc-srap/issues/103#issuecomment-4878025815

Goal for today's call: approve a text that can be posted for public comment.

Change sentence to "schema.org provides a useful set of ... "

Karen: People (potential users of SRAP) assume that they should understand what the text says and what they are supposed to do, and not everybody knows (about) SKOS. Nothing in DC terms says SKOS.

Tom: DC Terms precedes SKOS.

Phil: Tend to agree with Karen. Concerned about what accessibility people may know. Most of that paragraph is about the subject property. See what Tom is trying to do - reference to past. But what is needed here: justify property in own right. Could say pretty much the same thing. How we expect accessibility to be used, not how subject used.

Karen: Agree.

Argie: How much rewriting needed?

Niklas: We also do not want to preclude use of property with things of other types.

Phil [typing out suggestion on fly]: 
1. dcterms:accessibility,  for use with URIs. We propose the creation of a property dcterms:accessibility which is intended to be used with URIs of concepts that express the accessibility of the resource. In practice, the values would often be SKOS concepts (or would be logically equivalent to skos:Concepts as far as RDF is concerned). This approach parallels how we believe dct:subject may be used.

Argie: How about "The URIs should be taken from controlled RDF vocabularies in standards such as SKOS."

__`dcterms:accessibility` for use with URIs__. We propose the creation of a property `dcterms:accessibility`, which is intended to be used with URIs identifying accessibility features or hazards of a resource. These accessibility features should be selected from controlled RDF vocabularies using standards such as SKOS. This approach parallels how we believe `dcterms:subject` may be used.

Voting at https://github.com/dcmi/dc-srap/issues/103#issuecomment-4878025815

Karen: Also, in 3. we mention schema.org too abruptly. Rather, say "such as, for example"

Karen: This emphasizes Schema.org too much.

Niklas: This is useful as a summary but perhaps not a way to lead the proposal.

Phil: People will ask about Schema.org.

Argie: Seems like we cannot publish now - may need to restructure.

Kalli: Perhaps we can point to the IFLA Accessibility Metadata Statement and Principles draft, even if for our own reference? https://repository.ifla.org/items/df77f7bd-72dc-4213-95a9-76bdfb97eaa5 (No vocabularies are specified at all)
